La demande
Présentation

CICAR est une agence de location automobile implantée aux Îles Canaries, opérant depuis plusieurs aéroports de l'archipel (Las Palmas de Gran Canaria, Tenerife Sud, El Hierro, La Palma, La Gomera…). L'agence dispose d'un siège social à Tenerife connecté à ses agences locales via un réseau WAN inter-sites.

Dans le cadre de son développement, CICAR a souhaité moderniser son infrastructure informatique pour améliorer la fiabilité de ses services internes et la disponibilité de son application web de réservation en ligne. L'entreprise ne disposait pas d'outils de supervision ni de gestion des incidents, rendant difficile la détection des pannes et augmentant les délais d'intervention depuis le siège.

Cette PPE porte sur l'agence d'El Hierro (Valverde), réseau 192.168.54.0/24, reliée au siège et aux autres agences distantes par des tunnels VPN IPsec inter-agences.

Votre responsable prévoit la mise en place d'une supervision centralisée afin d'avoir une vue d'ensemble du parc machine et logiciel installé dans l'agence, ainsi qu'un logiciel de traitement des tickets d'incidents pour remonter les problèmes rencontrés par les utilisateurs. Les produits retenus sont Centreon (supervision SNMP), GLPI et son agent (gestion du parc et des tickets). Il vous appartiendra de rédiger les procédures nécessaires à la déclaration d'un ticket et de justifier du mode choisi pour la création du ticket.

Objectifs

Éléments à fournir :

  • Configuration de l'accès distant sécurisé MobaXterm via VPN IPsec
  • Activation du service SNMP sur les serveurs Linux de l'agence
  • Intégration des équipements dans Centreon (supervision depuis le siège Tenerife)
  • Définition des seuils d'alerte (CPU > 80%, espace disque < 10%)
  • Déploiement de GLPI + GLPI_Agent sur les postes et serveurs
  • Rédaction des procédures pour la création des tickets d'incidents
  • Mise en place du VPN IPsec sécurisé et du pare-feu Stormshield SNS 310
  • Activation du MFA pour les accès administrateurs
  • Tests de supervision : simulation de pannes, vérification des alertes
  • Mise à jour du schéma de l'infrastructure réseau et du plan d'adressage

Configuration de l'accès distant MobaXterm via VPN IPsec

Un tunnel VPN IPsec est établi entre le Stormshield SNS 310 de l'agence El Hierro et le pare-feu du siège de Tenerife. Cela permet depuis le poste d'administration du siège d'accéder en SSH à l'ensemble des VMs Linux de l'agence via MobaXterm. Le service OpenSSH est activé sur SRV-EL-MAIL, SRV-EL-WEB-01, SRV-EL-WEB-02 et SRV-EL-PROXY.

Activation SNMP sur les serveurs Linux de l'agence

Le service SNMP est installé et configuré sur l'ensemble des VMs Ubuntu de l'agence. La communauté public est restreinte à l'IP de Centreon (192.168.57.50) pour la remontée des métriques via le tunnel VPN IPsec. Étant limités par le nombre de sondes, seuls certains serveurs sont supervisés afin que toutes les agences et le siège puissent bénéficier du service. 

# Activation SNMP sur Ubuntu (SRV-EL-WEB-01 — 192.168.54.15)
sudo apt install snmpd -y

sudo nano /etc/snmp/snmpd.conf
# Ajouter :
rocommunity public 192.168.57.50
agentAddress udp:161

sudo systemctl enable snmpd && sudo systemctl start snmpd
Intégration des équipements dans Centreon

Centreon est installé au siège Tenerife (192.168.57.50). Les équipements de l'agence sont ajoutés comme hôtes supervisés. Les métriques remontent via SNMP v2c à travers le tunnel VPN IPsec. Des seuils d'alerte sont définis : CPU > 80% → alerte, espace disque < 10% → alerte critique.

GLPI est lié à l'Active Directory el-cicars.es via LDAP pour l'authentification des utilisateurs du domaine.

Procédure de création de ticket par un Self-Service
  1. Étape 1  — Ouvrir une session avec les identifiants du domaine el-cicars.es
  2. Étape 2  — Sélectionner la catégorie d'incident
  3. Étape 3  — Évaluer le niveau d'urgence de « Très basse » à « Très haute »
  4. Étape 4  — (facultatif) Ajouter un élément associé à l'incident, un PC par exemple
  5. Étape 5  — Rentrer le nom du technicien qui a observé le problème dans le champ « Observateur »
  6. Étape 6  — Donner un titre à l'incident
  7. Étape 7  — Écrire la description la plus détaillée de l'incident et y glisser, si besoin, des fichiers preuves
  8. Étape 8  — Pour finir, appuyer sur « Soumettre la demande »

Haute disponibilité — HAProxy + serveurs Apache en miroir

HAProxy est déployé en DMZ (192.168.64.1) et répartit les requêtes HTTP/HTTPS en round-robin vers les deux serveurs Apache (WEB-01 à .15 et WEB-02 à .16). Les deux instances partagent la même base de données MariaDB répliquée. Les certificats SSL sont gérés via Let's Encrypt. 

# /etc/haproxy/haproxy.cfg
frontend http_front
    bind *:80
    default_backend web_servers

backend web_servers
    balance roundrobin
    option httpchk HEAD / HTTP/1.0
    server web01 192.168.54.15:80 check
    server web02 192.168.54.16:80 check

Sauvegardes Veeam vers NAS RAID-5 du siège

Les sauvegardes des VMs sont orchestrées par Veeam Backup & Replication installé au siège. Les snapshots ESXi sont transférés via le VPN IPsec vers le NAS RAID-5 du siège Tenerife, en manuel sur les postes utilisateurs.
