PPE 2 – Infrastructure Web & Haute Disponibilité

Mise en œuvre d'une architecture 3-tiers résiliente pour le système de réservation CICAR.

1. Analyse du besoin

Dans le cadre du projet EL-CICAR, la disponibilité du service de location est critique. Toute interruption du serveur web entraîne une perte directe de chiffre d'affaires. Le PPE 2 consiste à déployer une infrastructure capable de supporter la perte d'un nœud applicatif sans impact utilisateur.

● Architecture

  • Virtualisation sous VMware ESXi
  • Segmentation LAN / DMZ
  • Répartition de charge (Layer 7)

● Services

  • HAProxy 2.6 (Load Balancer)
  • Serveurs Apache2 (Nodes)
  • Base MariaDB Master-Slave

2. Configuration du Répartiteur (HAProxy)

Le serveur SRV-EL-PROXY agit comme point d'entrée unique en DMZ. Il analyse la santé des serveurs en arrière-plan et distribue les requêtes HTTP/HTTPS.

🌐
PHOTO 1 — Configuration HAProxy
Définition du Frontend (port 80) et du Backend composé de la ferme de serveurs web.
Algorithme utilisé : Round-Robin avec persistance de session.
haproxy_config.png 
# Extrait de /etc/haproxy/haproxy.cfg
backend web-cluster
    balance roundrobin
    option httpchk GET /check.php
    server WEB-01 192.168.54.15:80 check inter 2000 fall 3 rise 2
    server WEB-02 192.168.54.16:80 check inter 2000 fall 3 rise 2

3. Redondance des données (MariaDB)

Pour que la haute disponibilité soit totale, les données de réservation doivent être présentes sur les deux serveurs. Une réplication asynchrone est mise en place entre WEB-01 (Master) et WEB-02 (Slave).
