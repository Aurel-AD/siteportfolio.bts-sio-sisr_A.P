La demande
Présentation

CICAR est une agence de location automobile implantée aux Îles Canaries, opérant depuis plusieurs aéroports de l'archipel (Las Palmas de Gran Canaria, Tenerife Sud, El Hierro, La Palma, La Gomera…). L'agence dispose d'un siège social à Tenerife connecté à ses agences locales via un réseau WAN inter-sites.

La CICAR est une jeune société de location de véhicule de tourisme en plein essor, implantée depuis 2012 dans les Îles Canaries. Elle possède une flotte de 65 véhicules répartis dans plusieurs agences sur les îles de Tenerife (qui est aussi le siège de l'entreprise), Lanzarote, Fuerteventura, La Palma, El Hierro, La Gomera et Gran Canaria.

Cette PPE porte sur l'agence d'El Hierro (Valverde), réseau 192.168.54.0/24, reliée au siège et aux autres agences distantes par des tunnels VPN IPsec inter-agences.

Suite à un incident de perte de connexion inter-îles ayant pénalisé l'activité de réservation plusieurs jours, le directeur de la CICAR souhaite externaliser les serveurs de réservation des véhicules, service actuellement hébergé sur les serveurs du siège. Chaque agence disposera désormais de ses propres serveurs locaux avec un mécanisme de load balancing (HAProxy en DMZ) et une sauvegarde automatique des VMs via Veeam Backup vers le NAS du siège.

Objectifs

Éléments à fournir :

  • Mise en place d'une architecture trois-tiers en agence (HAProxy DMZ → WEB-01/WEB-02 LAN → MariaDB)
  • Déploiement de deux serveurs Debian (WEB-01 et WEB-02) avec Apache, PHP, MariaDB et Adminer
  • Configuration du Load Balancing HAProxy en DMZ (192.168.64.1) en round-robin
  • Mise en place de la réplication MariaDB entre WEB-01 et WEB-02 (miroir)
  • Configuration du pare-feu Stormshield SNS 310 (zones LAN / DMZ / WAN)
  • Déploiement et configuration de Veeam Backup & Replication
  • Sauvegarde distante quotidienne des VMs vers le NAS RAID-5 du siège
  • Sauvegarde des paramètres des équipements d'interconnexion (SNS310, Switch, AP WiFi)
  • Tests de basculement et de continuité de service
  • Mise à jour du schéma de l'infrastructure réseau et du plan d'adressage

Le projet

Descriptif détaillé des actions réalisées pour la mise en place du projet d'externalisation des serveurs de réservation, de load balancing et de sauvegarde automatique de l'infrastructure de l'agence CICAR El Hierro.

Mise en place de l'architecture trois-tiers

L'architecture trois-tiers est déployée en agence : les requêtes internet arrivent sur le Stormshield SNS 310 (WAN), sont routées vers HAProxy dans la DMZ (192.168.64.1), qui répartit la charge en round-robin vers les deux serveurs Apache WEB-01 (192.168.54.15) et WEB-02 (192.168.54.16) situés dans le LAN. Les deux instances partagent la même base de données MariaDB répliquée.

Configuration du Load Balancing HAProxy

HAProxy est installé sur une VM Debian en DMZ (192.168.64.1). Il écoute sur les ports 80 et 443 et répartit les requêtes HTTP/HTTPS en round-robin vers WEB-01 et WEB-02. Des health checks sont activés pour détecter automatiquement une panne d'un backend et rediriger tout le trafic vers le serveur disponible. 

# /etc/haproxy/haproxy.cfg — Configuration Load Balancer
global
    log /dev/log local0
    maxconn 2000

defaults
    mode http
    log global
    option httplog
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend http_front
    bind *:80
    bind *:443 ssl crt /etc/ssl/certs/cicar.pem
    redirect scheme https if !{ ssl_fc }
    default_backend web_servers

backend web_servers
    balance roundrobin
    option httpchk HEAD / HTTP/1.1\r\nHost:\ localhost
    server web01 192.168.54.15:80 check inter 2s rise 2 fall 3
    server web02 192.168.54.16:80 check inter 2s rise 2 fall 3

listen stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 30s

Déploiement des serveurs Web Debian (WEB-01 et WEB-02)

Les deux serveurs de réservation sont installés sous Debian en ligne de commande (sans interface graphique) afin de limiter les surfaces d'attaque. Chaque serveur embarque la stack complète : Apache 2.4, PHP 8.x, MariaDB et Adminer (interface web d'administration de la base de données). Les deux instances sont en miroir avec réplication de la base MariaDB. 

# Installation de la stack LAMP sur Debian (WEB-01 : 192.168.54.15)
apt update && apt upgrade -y

# Installation Apache + PHP + MariaDB
apt install -y apache2 php php-mysql php-curl php-gd mariadb-server

# Installation Adminer (gestion base de données en ligne)
mkdir /var/www/html/adminer
wget https://github.com/vrana/adminer/releases/latest -O /var/www/html/adminer/index.php

# Sécurisation MariaDB
mysql_secure_installation

# Activer les services au démarrage
systemctl enable apache2 mariadb
systemctl start  apache2 mariadb

Réplication MariaDB — Miroir WEB-01 → WEB-02

La réplication maître-esclave MariaDB est configurée entre WEB-01 (maître, .15) et WEB-02 (esclave, .16). Toute modification de la base de données sur WEB-01 est automatiquement répliquée sur WEB-02, garantissant la cohérence des données et la continuité de service en cas de panne d'un des serveurs. 

# Sur WEB-01 (maître) — /etc/mysql/mariadb.conf.d/50-server.cnf
[mysqld]
server-id       = 1
log_bin         = /var/log/mysql/mysql-bin.log
binlog_do_db    = cicar_reservation

# Créer l'utilisateur de réplication
mysql -u root -p
CREATE USER 'replicator'@'192.168.54.16' IDENTIFIED BY 'Afpi5962!';
GRANT REPLICATION SLAVE ON *.* TO 'replicator'@'192.168.54.16';
FLUSH PRIVILEGES;

# Sur WEB-02 (esclave) — /etc/mysql/mariadb.conf.d/50-server.cnf
[mysqld]
server-id = 2

# Configurer la réplication
CHANGE MASTER TO
  MASTER_HOST='192.168.54.15',
  MASTER_USER='replicator',
  MASTER_PASSWORD='Afpi5962!',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=0;
START SLAVE;

# Vérifier la réplication
SHOW SLAVE STATUS\G

Tests de basculement — Continuité de service

Sauvegardes automatiques Veeam vers NAS RAID-5 du siège

Les sauvegardes des VMs ESXi de l'agence sont orchestrées par Veeam Backup & Replication installé au siège de Tenerife. Les snapshots ESXi (WEB-01, WEB-02, SRV-EL-PROXY, AD) sont transférés chaque nuit via le tunnel VPN IPsec vers le NAS RAID-5 du siège. Un job de sauvegarde quotidien est configuré avec une rétention de 7 jours.

Sauvegarde des paramètres des équipements d'interconnexion

Les configurations des équipements d'interconnexion sont exportées et sauvegardées afin de permettre une restauration rapide en cas de défaillance matérielle. Le Stormshield SNS 310, le Switch TP-Link SG-3424 et le point d'accès WiFi WA901-N font l'objet d'une sauvegarde de configuration régulière. 

# Sauvegarde configuration Stormshield SNS 310
# Via interface web : Maintenance → Sauvegardes → Télécharger la configuration
# Accès : https://192.168.54.254 → admin / Afpi5962!

# Sauvegarde configuration Switch TP-Link SG-3424
# Via interface web : System → System Tools → Backup
# Accès : http://192.168.54.20 → admin / Afpi5962

# Sauvegarde via CLI SSH (Switch TP-Link)
